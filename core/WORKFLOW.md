## Principles
- **P1 — verification over output.** Human attention is scarce; spend it verifying. Gate ∝ **blast-radius × irreversibility**, never effort.
- **P2 — intelligence-ordering (IO; "the librarian rule").** Intelligence flows downhill, never capped uphill. A lower layer (deterministic program *or* cheaper model) MAY locate / classify / verify-invariants / enforce-safety-bounds / surface-options+evidence; it MUST NOT decide the substance for, or limit what's concluded by, a higher-intelligence layer downstream. Sharp line: **enforce a mechanical invariant — yes; make the semantic decision — no.** Its outputs (labels/buckets/routes/filters) are advisory inputs the higher layer may override against live context, never binding. (*Librarian:* a cheap labeler assigning a *semantic attribute* at ingest capped the query model at the labeler's ceiling and measured badly wrong → store a neutral presence flag, let the smart layer judge.) IO is the **root** of exact-provenance, no-fabricated-answers, and sensors-not-actuators.
- **P3 — decorrelate generation, not just review.** The blind-spot logic that makes a same-family *review* panel "miss alike" applies equally to *generation*: a hard or open-ended task run always on one model inherits that model's consistent priors. Where idea-diversity is the value (hypothesis generation, contrarian analysis, hard open problems), **vary the generating model across families** on the same externalized context and **synthesize across the lenses** — parallel multi-lens on the highest-stakes items, a rotating single model on routine ones, cheap-pinned where diversity buys nothing (mechanical work). Two guardrails keep it honest: **(1) pin the decider** — diversify the *inputs* (generation/surfacing) but keep the *deciding* seat on one consistent frontier model (coherence + calibration integrity; a rotating judge corrupts its own scorecard); **(2) a contributing family reviews at REDUCED decorrelation — additive, never a substitute.** Decorrelation has two axes: *posture* (is it defending its own work?) and *family* (shared weight-priors). A **cold/blind thread** (fresh context, raw artifact only, no memory of having generated it) recovers *posture* but **not** *family* — and for a fresh file-in/file-out gate posture is **ambient** (every seat is already cold), so a contributor-cold seat mostly keeps the axis you had for free and loses the *family* axis a cross-family gate exists for. So a contributing family (the **set** of contributors — authors AND surfacers) may serve **only cold**, recorded **reduced (same-family)**, as an **additive** read — **never the sole independent basis** of a critical verdict. **A critical / autonomous gate (no per-verdict Owner spot-check) keeps a hard family floor: ≥1 full-independent (non-contributing) seat is REQUIRED and it FAILS CLOSED** — if every eligible family contributed, the artifact is *not gradable at that tier* → downgrade / block / escalate to the Owner, never a full verdict resting only on reduced seats. Two more integrity conditions when a contributor is seated cold: **re-author its framing by a non-contributor** first (a cold seat still re-reads its own hypothesis/advocacy if the packet carries it — the framing-trap), and **capture the contributing-set out-of-band** where the machinery must stand alone (a self-metered set is gameable *by omission* — hiding a surfacing would upgrade a reduced seat to a fake-pristine one; the fail-closed floor is the load-bearing control, best-effort metering + Owner spot-check only where an Owner checks each verdict). This constrains the *decorrelating lens only*, **not** the same-family coverage panel, which still runs — it composes with, never replaces, the review-side ladder. **Decorrelation is a shared budget:** a full-independent K-lens gate wants K families that did *not* contribute; if generation would spend them, **route generation to fewer families (freeing full seats)** rather than lean on reduced seats. (The *decider* MAY share a generator's family: generation↔decision decorrelation is redundant once the *decision* is independently checked by a cross-family gate — the gate decorrelates the decision, not the seating chart.) Enabler: **externalized state** (disk/store, not weights/context) makes the model a swappable *lens* over persistent memory. Cost-gated — parallel multi-lens is a high-stakes lever, not every run; P1 still rules (spend intelligence on blast-radius, not on everything).
- **Threat-model calibration:** gates are seatbelts for cooperative-but-fallible agents, not intrusion detection. Hostile-evasion hardening requires explicit Owner scope for an external/adversarial security threat.

## Roles *(bound to actual models/people in `core/BINDINGS.md`)*
**Owner** — human; steers, sets intent, ratifies tier, authorizes destructive acts, runs prod writes, and may explicitly override the PM on intent or risk acceptance; the Owner cannot relabel an uncontained mechanically confirmed invariant failure as passed. **Project Manager (PM)** — the pinned frontier LLM in the active build thread; owns phase boundaries and decides the disposition of every review finding. **Builder** — the executing agent; it may be the same frontier thread as the PM, but that does not count as independent review. **Gather agent** — subordinate; acquires context (recon/archaeology/prior-art), returns conclusions + source pointers; non-authoring. **Cold reviewer** — fresh, blind agent (same family as the Builder); reviews an artifact with no knowledge of how it was built. **Cross-family lens** — a reviewer from a *different model family*; its value is **decorrelated blind spots** (a same-family panel, however fresh, misses alike). **Free-roaming adversary** — a reviewer (any family) given *only* the redacted contract + the code (no prior findings, no rationale, no verdicts), tasked to break it any way **and to attack the contract itself**; the posture-decorrelation spine. **External gate** — an independent external review at major/critical. **Artifact store** — versioned dir for designs/tickets/verdicts.

*Delegation — how subagents **Gather / Review / Author** within the tiers (changing no tier and no gate), plus the **Retention guarantee** that keeps live-path/T2/T3 authoring in the Builder's own context — is `core/OPERATE.md` § Delegation.*

## Steer — classify before building
**Reversible** = a *tested, named restore-to-prior-state handle* exists before the write AND the Owner confirms it at classify-time (snapshot id · verified down-migration · delete-by-key *for pure INSERTs* — on an UPDATE it deletes rather than restores prior state, so a mutating write needs a snapshot/down-migration or is T3). A raw, unverified backup/export does **not** count. No confirmed handle ⇒ **irreversible**; Builder proposes **T3** until the Owner confirms a handle to downgrade.

**Decision tree — first match, evaluate 1→4:**
1. Executes (or deploys code that will execute) an **irreversible** write to live prod → **T3 Critical**
2. A **reversible** write to live prod, OR changes **live-behavior-path** code, OR changes **chain/stateful control logic** → **T2 Major**
3. **Local — touches no live prod** (a test, a local tool/script, a local-DB mutation, a *bounded* read-only prod query) → **T1 Minor** *(an **unbounded** prod read is **T3** — OOM risk; download an artifact instead)*
4. **Local, non-code** — docs, comments, a log string, a memory pointer → **T0 Trivial**

Builder proposes the tier; **Owner ratifies before any T2/T3 gate.** A change bundling separable tiers → split into separately-gated commits; only when inseparable does the whole take the highest tier. The tier sets a **code gate** (review depth) + an **execution gate** (run authorization, only if it runs a prod write).

| Tier | Code gate | Execution gate *(only if prod write)* |
|---|---|---|
| **T0** | self-check → proceed | — |
| **T1** | one blind cold reviewer → proceed | — |
| **T2** | cold panel → cross-family lens [if avail] → external → Owner push-GO | named-GO (may cover a bounded batch) + execution protocol |
| **T3** | cold panel → **both** cross-family families (lens **required** + external) → Owner push-GO | **named-GO per write** + execution protocol |

*Tier table = local build/run depth; any code **push** also needs an Owner push-GO (see "Pushing is a separate axis").*
**The tier that selects the row is the DECLARED one, Owner-ratified per above** — a declaration is a self-report, and no mechanism can tell a correctly-tiered change from a mis-tiered one. **Escalate mid-task freely; lowering mid-task requires the Owner's confirmed reversibility handle (above) and is recorded in the PM disposition.** *(This binds the PM and Owner — **no mechanism enforces it**; a hook cannot tell a dodge from an Owner-authorised correction. `.claude/hooks/guard-gate-ladder.mjs` only surfaces the declared tier's ladder, failing closed to T3, and never denies. Claude lane only.)*

**Autonomy local, boundaries central:** T0/T1 the Builder builds and runs locally free; T2 adds an external adversary; T3 stops at the Owner.

**Pushing is a separate axis from the tier** (the push *is* the deploy where the target auto-deploys): **docs-only (T0) pushes without a GO (unless the branch carries unpushed code — a push ships the whole branch); any push containing code needs an Owner push-GO regardless of tier** — T1 code: cold + push-GO; T2/T3: cold + external + push-GO. A tier governs build/local-run depth; reaching the auto-deploy target is always re-gated.

**Core-document amendments are never T0** (no prod blast-radius, but high leverage — a doctrine/architecture flaw propagates to every change under it). Gate = **blind cold pass** (panel ≥2 for a substantial change; a single reviewer for a minor clause) → **Owner sign-off on the wording**; **no external gate** (no code to verify; an idea's substance is the Owner's call). The Owner wording sign-off also stands as its push-GO. *Exception:* an amendment carrying **executable content** (schemas, an API contract code depends on) follows the **code tiers** for that portion. **Amendments to the gate machinery itself** — *any clause that sets a tier, a gate trigger, a NO-GO condition, a reviewer payload, an invariant, or a mitigation/manifest schema (if in doubt, it is)* — require an **adversarial walk-through** — ≥1 reviewer constructs a concrete scenario where an agent technically *complies* yet the targeted failure still occurs — before sign-off; design-mode prose-reading alone is insufficient. Core docs are listed in `core/BINDINGS.md`.

**Before building:** recommend the intelligence tier; for T2/T3 write **intent + falsifiable acceptance criteria** as a design doc *before* coding; for chain/stateful logic name the **adversarial cases up front** (crash-mid-sequence · retry · corrupt-state→fail-closed · starvation · empty-tick) and write them as tests before the gate.

## Gate — the review ladder
*The ladder itself is below. **How a review is constructed and judged** — the cold-review payload contract, decorrelation, the cross-family lens, and the external gate — is `core/REVIEW.md`. **Tool invocation** (Codex / Gemini runners, model-effort matrix, receipts, traps) is `core/GATES.md`.*

**Gate states.** FAIL → stop the gate → PM verifies and dispositions each finding. `REMEDIATE` → fix → scoped bookend; `DEFER` / `DECLINE` → record and do not edit; `ESCALATE` → Owner. Cold vs external disagreement is evidence for PM adjudication, not automatic severity-maxing. Mis-tier discovered → escalate and restart the required gates.

**A verdict is INPUT, not an instruction; a ladder ends on a JUDGMENT, never on reviewer silence** — an adversarial reviewer never runs out of findings, so *"no findings left"* is unreachable and must never be an exit condition. **Emit the disposition after EVERY verdict, before any edit; no emission ⇒ no fix** (an undispositioned fix is out-of-process — revert it):
> `GATE ROUND <n>/3 · changeset <name> · verdict <GO|NO-GO>` — per finding: **REAL?** *(does it occur, in the **stated** threat model? reachable?)* · **SCOPE?** *(the FEATURE asked for, or machinery **you** introduced?)* · **BOUNDED?** *(small and final, or does it mint new surface?)* · **WORTH IT?** *(what breaks if you DECLINE?)* → **REMEDIATE | DEFER | DECLINE | ESCALATE** + reason. **A `REMEDIATE` on CODE must carry a `TRIGGER:` line — a concrete reachable path in the *declared* threat model (inputs/state → the bad outcome, runnable or directly derivable from the diff); no trigger ⇒ `DEFER`/`DECLINE`, and a REMEDIATE lacking one is out-of-process (revert the fix).** **Symmetric brake — a `DEFER`/`DECLINE` of a reviewer-rated BLOCKER/HIGH owes the FAILED trigger: the strongest trigger you could build + the specific reason it is out-of-model / unreachable / an accepted residual — never bare absence, and never "the Owner didn't recognise it" (verify reachability against the code). Precedence still forbids deferring a *confirmed* material failure.** *(= instruction-physics "reproduce before folding" promoted to code — `core/REVIEW.md`; composes with Precedence below — a mechanically-confirmed failure already HAS its trigger, so it still REMEDIATEs.)* Close: `LADDER: continue | STOP-AND-ESCALATE`. After round 3 add `PAST-SOFT-STOP: CONTINUE | STOP-AND-ESCALATE` + a specific `JUSTIFICATION` if continuing.

**`DECLINE`/`DEFER` are first-class, and ALL-REMEDIATE forces a self-audit:** a round whose findings are *every* `REMEDIATE` (>1 finding) pauses the ladder for a mandatory **in-thread self-audit** — re-read the threat-model contract and re-verify each `TRIGGER` actually holds; continue on a clean audit, and **escalate the scope to the Owner ONLY if a trigger does not hold or scope is drifting.** A clean multi-bug round is normal (a first cold panel routinely finds several real bugs) and does NOT interrupt the Owner — this is the backstop for a fabricated `TRIGGER`, not a tax on real bugs. **Soft stop after 3 NO-GO rounds per CHANGESET.** Round 4+ may continue without Owner presence only after the PM emits the repeated warning plus a specific justification naming what changed, why another cycle is warranted, and why it is bounded rather than scope churn. **Rounds count per changeset — the task, not the file version:** re-freezing after a fix, splitting findings, or swapping reviewer/family does **not** reset the count. **Stop earlier on any SCOPE signal:** findings arriving *in code you added last round* · the fix needs a contract edit · unreachable in the threat model, unfixable at this layer, or a nit. **Declare the threat model + accepted residuals BEFORE round 1 and hand them to the reviewer** (`core/REVIEW.md` contract); **undeclared ⇒ the gate's first act is declaring it with the Owner** (Owner-ratified like the tier — an inflated/adversarial model needs explicit Owner scope, so a fabricated `TRIGGER` naming an out-of-model actor has nothing to stand on). A finding outside that model or inside the residual set is `DEFER` by construction, not re-adjudicated (gates are seatbelts for cooperative agents, not intrusion detection — Threat-model calibration, P). **A false premise stops the work:** report it; never silently reinterpret and build. *(`<n>` is a CONVERSATION fact — verdicts dispositioned. **No hook counts it**; a round-counting hook was retired 2026-07-18. Origin: `docs/journal/workflow_v2_phase3_design.md`.)*

**PM finding disposition — reviewers surface; the frontier PM decides.** A review verdict stops the current gate, but it is not an automatic work order. Before editing, the PM records each finding as: **REMEDIATE** (confirmed or reproducible and material to the contract, acceptance criteria, or a required gate dimension); **DEFER** (real but non-material to this release; named backlog/ticket); **DECLINE** (false positive, unsupported speculation, cosmetic nit, or disproportionate non-material edge case); or **ESCALATE** (requires Owner intent/risk judgment). Severity is evidence, not authority: the PM verifies the claim, weighs likelihood × impact × remediation cost, and owns the release-level verdict. **Precedence:** a mechanically confirmed invariant failure or material correctness, security, safety, provenance, reversibility/blast-radius, deploy-order, contract/additive, IO, or mitigation-integrity failure cannot be `DEFER`red or `DECLINE`d; it requires `REMEDIATE`, or `HOLD` + `ESCALATE`. The only risk-acceptance path remains `HOLD—OWNER-ACCEPTED CONTAINMENT` (default-off flag · quarantine · non-prod), never a passed gate.

**Final-gate rule — the last material edit is never the only unreviewed edit.** Every gate reviews the *pre-fix* diff. After PM triage, only **REMEDIATE** findings are fixed. The approved remediation delta receives one cross-family bookend when available — *code* in diff-mode; *design/core-doc* in design-mode on the full updated artifact — or one same-family cold re-review when the lens is unavailable. The bookend checks approved fixes and regressions; unrelated new findings return to PM triage and are **not** auto-remediated. A structural/substantial approved fix restarts the external gate only when the PM classifies it as such; after the soft stop, that cycle also needs the continuation justification. A clean pass, or one containing only permissible declined/deferred items, closes the ladder.

**Gate continuation — soft stop, never clean-pass chasing.** The initial final gate plus **two PM-approved remediation/bookend cycles** is the target. The PM may stop earlier for diminishing returns. At round 4+ it either stops/escalates or emits `PAST-SOFT-STOP: CONTINUE` plus the required justification; there is no Owner-presence hard stop. Remaining nits and non-material edges are declined or ticketed alongside `GO`. Unresolved structural or Owner-judgment issues terminate as `HOLD + ESCALATE`. An unresolved BLOCKER/HIGH in a required dimension cannot ship as a passed gate — at most `HOLD—OWNER-ACCEPTED CONTAINMENT` on explicit written acceptance, never relabeled `GO`.

**Shipping — any code push.** Any push **containing code** to the live / auto-deploy target needs the **code gate + a named Owner push-GO** (T1: cold + push-GO; T2/T3: cold + external + push-GO) — never a routine code push. **Push-GO follows TIER, not file type: docs-only pushes are free at T0 only — a T1–T3 push (code OR instruction; a prod-write-directing doc like the Runbook tiers by what it directs) needs its tier's gate + the Owner push-GO.** If the pushed code carries a prod-write path, the push-GO authorizes the *deploy* only; each prod write it then performs takes its own named **write-GO** (the two are never merged).
